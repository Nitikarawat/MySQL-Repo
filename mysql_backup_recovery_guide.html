Core Concepts — Why Backup?

As a Senior DBA, the first rule is: a backup you haven't tested is not a backup — it's a hope. In production, data loss events are not a matter of "if" but "when." Understanding the why, what, and how of backups is your most critical responsibility.

💥

Hardware Failure

Disk corruption, RAID failure, server meltdown — hardware dies. Backups are your safety net.

🧑‍💻

Human Error

The #1 cause of production incidents: DROP TABLE orders; without a WHERE clause. It happens.

🦠

Ransomware / Corruption

Malware that encrypts or destroys your data. Isolated, tested backups are your only defense.

🔄

Migration / Upgrade

Before any major schema change or MySQL version upgrade, a full backup is non-negotiable.

🔍

Auditing / Compliance

Regulatory requirements (GDPR, HIPAA, PCI-DSS) often mandate data retention and recovery capabilities.

📊

Replication Bootstrap

Setting up a new replica requires a consistent snapshot of the primary — mysqldump provides exactly that.

Key Terminology

TermDefinitionProduction Context
RPORecovery Point Objective — max acceptable data loss (in time)"We can lose at most 1 hour of data" → hourly backups
RTORecovery Time Objective — max time to restore service"System must be back in 4 hours" → dictates your restore strategy

Types of Backups

Full Backup

A complete snapshot of all data at a point in time. The foundation of any backup strategy. All recovery chains start from a full backup.

Incremental Backup

Backs up only changes since the last backup (full or incremental). With mysqldump, this is achieved via binary log rotation. Requires all binlogs in the chain to restore.

Differential Backup

Backs up all changes since the last full backup. Simpler restore (full + one differential) but grows larger each day.

Production Standard: Weekly full backup + daily incremental via binary logs + transaction-level PITR. For critical systems, combine mysqldump with Percona XtraBackup for the best of both worlds.

03

mysqldump — The Utility

mysqldump is a command-line client utility for performing logical backups. It generates a SQL script containing all the statements needed to recreate the database objects (tables, views, procedures, etc.) and repopulate them with data.

How mysqldump Works Internally

Understanding the internals makes you a better DBA — not just a command runner.

⚙️

mysqldump connects to MySQL, issues FLUSH TABLES WITH READ LOCK (for MyISAM) or uses START TRANSACTION WITH CONSISTENT SNAPSHOT (for InnoDB), reads all data via SELECT, and writes it as INSERT statements. For InnoDB, it holds a single transaction snapshot — allowing other writes to continue during the dump.

Installation & Verification

bash · Server Setup CheckPRODUCTION
# Verify mysqldump is installed and check version
mysqldump --version
# → mysqldump  Ver 8.0.35 Distrib 8.0.35, for Linux (x86_64)

# Check MySQL server version (must match or be compatible)
mysql --version

# Verify credentials work before any backup script
mysql -u root -p -e "SELECT VERSION(), NOW();"

# Check that binary logging is enabled (needed for PITR)
mysql -u root -p -e "SHOW VARIABLES LIKE 'log_bin';"
# Variable_name | Value
# log_bin       | ON     ← this must be ON in production

Basic Syntax Structure

Syntax Reference
mysqldump [OPTIONS] [database_name] [table1 table2 ...]  >  output_file.sql

# The three main patterns:
# 1. Single database
mysqldump -u username -p database_name  > backup.sql

# 2. Multiple specific databases
mysqldump -u username -p --databases db1 db2 db3  > backup.sql

# 3. ALL databases on the server
mysqldump -u username -p --all-databases  > backup.sql

Your First Production Backup

bash · First BackupPROD
# Step 1: Create a dedicated backup directory
mkdir -p /var/backups/mysql/$(date +%Y-%m-%d)

# Step 2: Create a .my.cnf file for passwordless auth (NEVER hardcode passwords in scripts!)
cat > /root/.mybackup.cnf << 'EOF'
[client]
user=backup_user
password=YourSecurePassword123!
host=127.0.0.1
port=3306
EOF
chmod 600 /root/.mybackup.cnf   # CRITICAL: only root can read this

# Step 3: Create a dedicated backup user with minimal privileges
mysql -u root -p -e "
  CREATE USER 'backup_user'@'localhost' IDENTIFIED BY 'YourSecurePassword123!';
  GRANT SELECT, SHOW VIEW, TRIGGER, LOCK TABLES, EVENT, RELOAD,
        REPLICATION CLIENT ON *.* TO 'backup_user'@'localhost';
  FLUSH PRIVILEGES;
"

# Step 4: Run your first backup using the config file
mysqldump --defaults-extra-file=/root/.mybackup.cnf \
  --single-transaction \
  --routines \
  --triggers \
  --events \
  --all-databases \
  > /var/backups/mysql/$(date +%Y-%m-%d)/full_backup_$(date +%H%M%S).sql

# Step 5: Verify the backup file was created and has size
ls -lh /var/backups/mysql/$(date +%Y-%m-%d)/
wc -l /var/backups/mysql/$(date +%Y-%m-%d)/full_backup_*.sql
⚠️

Production Rule: Never use -p inline in scripts — it exposes your password in shell history and process lists. Always use --defaults-extra-file with a permission-restricted config file.

Flags & Options — Deep Dive

Understanding every important flag separates a junior DBA from a senior one. Here are the flags you must know with production context for each.

Consistency & Transaction Flags

Consistency OptionsCRITICAL
# --single-transaction (THE most important InnoDB flag)
# Starts a transaction before dumping. InnoDB provides a consistent snapshot.
# Other writes CAN continue. NO table locks for InnoDB tables.
--single-transaction

# Example: Full backup without downtime (InnoDB only)
mysqldump --defaults-extra-file=/root/.mybackup.cnf \
  --single-transaction \
  --all-databases \
  > /backups/full.sql

------------------------------------------------------------

# --lock-tables (Default behavior for MyISAM)
# Locks ALL tables in each database before dumping it.
# This BLOCKS writes! Avoid for production InnoDB workloads.
--lock-tables          # ON by default, disable with --skip-lock-tables
--skip-lock-tables     # Use when --single-transaction is sufficient

------------------------------------------------------------

# --lock-all-tables (for mixed engine environments)
# Issues FLUSH TABLES WITH READ LOCK — locks every table server-wide
# Ensures consistency across ALL storage engines (InnoDB + MyISAM)
--lock-all-tables

# ⚠️  PRODUCTION WARNING: --lock-all-tables holds global lock for entire dump duration
# On a 100GB database, this could lock production for 30+ minutes!

Binary Log Position Flags

Binary Log OptionsPITR & REPLICATION
# --master-data (deprecated in 8.0, replaced by --source-data)
# --source-data=1  → adds CHANGE MASTER TO statement (uncommented)
# --source-data=2  → adds CHANGE MASTER TO as a COMMENT (informational)

mysqldump --defaults-extra-file=/root/.mybackup.cnf \
  --single-transaction \
  --source-data=2 \           # Records binlog filename + position as comment
  --all-databases \
  > /backups/full_with_binlog_pos.sql

# Check what was recorded in the dump file:
head -50 /backups/full_with_binlog_pos.sql | grep -i "MASTER"
# Output: -- CHANGE MASTER TO MASTER_LOG_FILE='mysql-bin.000042', MASTER_LOG_POS=4891620;
# This tells you: "to do PITR, apply binlogs starting from this position"

------------------------------------------------------------
# --flush-logs
# Flushes binary logs before dump — creates a clean new binlog file.
# Makes PITR simpler: all changes AFTER dump start in a fresh binlog.
--flush-logs

# Combined best practice for PITR-ready backup:
mysqldump --defaults-extra-file=/root/.mybackup.cnf \
  --single-transaction \
  --source-data=2 \
  --flush-logs \
  --all-databases \
  > /backups/pitr_ready_$(date +%Y%m%d_%H%M%S).sql

Content Control Flags

What to Include in the Backup
# Include stored procedures and functions
--routines    # (or -R)

# Include triggers (INCLUDED by default but good to be explicit)
--triggers

# Include scheduled events
--events      # (or -E)

# Schema only — no data (useful for dev environment setup)
--no-data     # (or -d)

# Data only — no CREATE TABLE statements (useful for data migration)
--no-create-info   # (or -t)

# Skip specific tables (e.g., logs, audit tables that are huge)
--ignore-table=mydb.audit_log \
--ignore-table=mydb.application_log

# Add DROP TABLE before each CREATE TABLE (safe for restore overwrite)
--add-drop-table   # Included by default — very important for clean restores

# Add DROP DATABASE before CREATE DATABASE
--add-drop-database

# PRACTICAL EXAMPLE: Schema-only backup for dev team
mysqldump --defaults-extra-file=/root/.mybackup.cnf \
  --no-data \
  --routines \
  --triggers \
  --events \
  --databases ecommerce_prod \
  > /backups/schema_only_$(date +%Y%m%d).sql

Performance Flags

Performance OptimizationLARGE DATABASES
# --compress (-C): Compress communication between client and server
# Useful for remote backups over network
--compress

# Pipe directly through gzip — ALWAYS do this for large databases
mysqldump --defaults-extra-file=/root/.mybackup.cnf \
  --single-transaction --all-databases \
  | gzip -9 \
  > /backups/full_$(date +%Y%m%d).sql.gz
# A 10GB database typically compresses to 1-2GB with gzip -9

# Use pigz for parallel compression (much faster on multi-core servers)
mysqldump --defaults-extra-file=/root/.mybackup.cnf \
  --single-transaction --all-databases \
  | pigz -9 \
  > /backups/full_$(date +%Y%m%d).sql.gz

------------------------------------------------------------
# --extended-insert (DEFAULT = ON)
# Groups multiple rows into single INSERT statements → smaller file, faster restore
# INSERT INTO t VALUES (1,'a'),(2,'b'),(3,'c');  ← extended (default)
--extended-insert     # ON by default
--skip-extended-insert  # One INSERT per row — use only for debugging/diffing

# --net-buffer-length & --max-allowed-packet
# For large BLOB/TEXT columns, increase packet size
--max-allowed-packet=512M

# --quick (-q): Retrieve rows one at a time instead of buffering
# CRITICAL for large tables — prevents memory exhaustion
--quick   # ON by default for mysqldump — just know this exists

The Gold Standard Production Backup Command

bash · Production Gold StandardUSE THIS IN PRODUCTION
#!/bin/bash
# The "gold standard" mysqldump command for InnoDB production databases

BACKUP_DIR="/var/backups/mysql"
TIMESTAMP=$(date +%Y%m%d_%H%M%S)
BACKUP_FILE="${BACKUP_DIR}/full_${TIMESTAMP}.sql.gz"

mysqldump \
  --defaults-extra-file=/root/.mybackup.cnf \     # secure credentials
  --single-transaction \                           # consistent InnoDB snapshot
  --source-data=2 \                                # embed binlog position
  --flush-logs \                                   # rotate binlog at start
  --routines \                                     # stored procedures
  --triggers \                                     # triggers
  --events \                                       # scheduled events
  --add-drop-database \                            # clean restore
  --add-drop-table \                               # overwrite existing tables
  --max-allowed-packet=512M \                      # handle large blobs
  --all-databases \                                # everything
  | pigz -9 \                                      # parallel compress
  > "${BACKUP_FILE}"

# Capture exit code BEFORE doing anything else
EXIT_CODE=$?

if [ $EXIT_CODE -ne 0 ]; then
  echo "BACKUP FAILED with exit code $EXIT_CODE" | mail -s "PROD BACKUP FAILURE" dba@company.com
  exit 1
fi

# Verify the compressed file is valid
pigz -t "${BACKUP_FILE}" && echo "Backup verified: ${BACKUP_FILE} ($(du -sh ${BACKUP_FILE} | cut -f1))"

You manage an e-commerce platform. Every night at 2 AM, you need a full backup of the ecommerce_db database with all objects.

🏭 Scenario 2 — Selective Table Backup PROD

A developer accidentally ran a wrong UPDATE on the products and inventory tables. You need a quick backup of just those tables before attempting a fix.

Scenario 2 — Specific Tables
# Backup specific tables only
mysqldump \
  --defaults-extra-file=/root/.mybackup.cnf \
  --single-transaction \
  ecommerce_db products inventory \    # db_name followed by table names
  > /backups/tables_products_inventory_$(date +%Y%m%d_%H%M%S).sql

# Verify specific table data
grep -c "^INSERT" /backups/tables_products_inventory_*.sql
# Should match your row counts from SELECT COUNT(*)
🏭 Scenario 3 — Backup Excluding Large Log Tables PROD

Your audit_log and email_queue tables have 500M+ rows and are not critical for DR. Excluding them cuts backup time from 4 hours to 20 minutes.

Scenario 3 — Exclude Tables
# Backup everything EXCEPT specific tables
mysqldump \
  --defaults-extra-file=/root/.mybackup.cnf \
  --single-transaction \
  --routines --triggers --events \
  --ignore-table=ecommerce_db.audit_log \
  --ignore-table=ecommerce_db.email_queue \
  --ignore-table=ecommerce_db.activity_sessions \
  --databases ecommerce_db \
  | gzip \
  > /backups/ecommerce_no_logs_$(date +%Y%m%d).sql.gz

# ⚠️  NOTE: --ignore-table still includes the CREATE TABLE statement!
# Only the data (INSERT statements) is skipped.
# To skip BOTH schema and data, use two separate mysqldump calls
# or post-process the SQL file.
🏭 Scenario 4 — Remote Server Backup to Local Storage PROD

Your backup server is different from your DB server. You need to pull backups remotely over the network securely.

Scenario 4 — Remote Backup
# Method 1: Direct remote connection (backup runs on backup server)
mysqldump \
  --defaults-extra-file=/root/.mybackup.cnf \
  --host=10.0.1.50 \                # DB server IP (internal network)
  --port=3306 \
  --single-transaction \
  --compress \                         # compress network traffic
  --all-databases \
  | gzip \
  > /backups/remote_full_$(date +%Y%m%d).sql.gz

------------------------------------------------------------
# Method 2: SSH tunnel (more secure — encrypts everything)
# On backup server, create SSH tunnel first:
ssh -L 3307:127.0.0.1:3306 dbserver@10.0.1.50 -N -f

# Then connect to tunnel (local port 3307 = remote MySQL)
mysqldump \
  --defaults-extra-file=/root/.mybackup.cnf \
  --host=127.0.0.1 \
  --port=3307 \
  --single-transaction \
  --all-databases \
  | gzip \
  > /backups/remote_ssh_tunnel_$(date +%Y%m%d).sql.gz
🏭 Scenario 5 — Backup for Replica Setup PROD

You need to add a new read replica to your production primary server. You need a backup with exact binary log position so the replica can sync from that point.

Scenario 5 — Replica Bootstrap Backup
# On the PRIMARY server — create replica seed backup
mysqldump \
  --defaults-extra-file=/root/.mybackup.cnf \
  --single-transaction \
  --source-data=1 \     # =1 means uncommented CHANGE MASTER TO (for replication setup)
  --flush-logs \
  --routines \
  --triggers \
  --events \
  --all-databases \
  | gzip \
  > /backups/replica_seed_$(date +%Y%m%d_%H%M%S).sql.gz

# Transfer to replica server
scp /backups/replica_seed_*.sql.gz replica-server:/tmp/

# ---- On REPLICA server ----
# Restore the backup
zcat /tmp/replica_seed_*.sql.gz | mysql -u root -p

# The --source-data=1 already embedded CHANGE MASTER TO in the dump.
# Configure replication:
mysql -u root -p << 'EOF'
CHANGE REPLICATION SOURCE TO
  SOURCE_HOST='10.0.1.50',
  SOURCE_USER='repl_user',
  SOURCE_PASSWORD='ReplPassword123!',
  SOURCE_AUTO_POSITION=1;  -- if using GTID
START REPLICA;
SHOW REPLICA STATUS\G
EOF

Restore & Recovery

Restoring is the true test of your backup strategy. In production, an untested restore is dangerous. Every backup should be test-restored on a staging server regularly.

Full Database Restore

Full Restore from mysqldumpDISASTER RECOVERY
# Method 1: Using mysql client with uncompressed file
mysql -u root -p < /backups/full_backup.sql

# Method 2: Compressed file (preferred in production)
zcat /backups/full_20240115_020000.sql.gz | mysql -u root -p
# or
gunzip -c /backups/full_20240115_020000.sql.gz | mysql -u root -p

# Method 3: With progress indicator (pv — pipe viewer)
# Install: apt-get install pv
pv /backups/full_backup.sql.gz | zcat | mysql -u root -p
# Shows: 450MiB 0:03:22 [2.22MiB/s] [=====>       ] 45%  ETA 0:04:05

------------------------------------------------------------
# Restore with verbose output and stop on first error
mysql \
  -u root -p \
  --verbose \
  -e "source /backups/full_backup.sql"

# Verify restore success
mysql -u root -p -e "SHOW DATABASES;"
mysql -u root -p -e "SELECT COUNT(*) FROM ecommerce_db.orders;"

Single Database Restore

Restore Single Database
# If the dump was created with --databases flag (includes CREATE DATABASE)
zcat /backups/ecommerce_db_backup.sql.gz | mysql -u root -p

------------------------------------------------------------
# If the dump was created WITHOUT --databases (no CREATE DATABASE statement)
# You must create the database first and specify it
mysql -u root -p -e "CREATE DATABASE IF NOT EXISTS ecommerce_db;"
zcat /backups/ecommerce_only.sql.gz | mysql -u root -p ecommerce_db

------------------------------------------------------------
# Restore to a DIFFERENT database name (useful for parallel validation)
mysql -u root -p -e "CREATE DATABASE ecommerce_db_restore_test;"
zcat /backups/ecommerce_db_backup.sql.gz \
  | sed 's/`ecommerce_db`/`ecommerce_db_restore_test`/g' \
  | mysql -u root -p

Single Table Restore — The Most Common Production Need

bash · Restore One Table from Full BackupMOST COMMON DR SCENARIO
#!/bin/bash
# Extract and restore a single table from a full backup
# Scenario: Developer ran "DELETE FROM orders WHERE 1=1" — restore orders table

DB_NAME="ecommerce_db"
TABLE_NAME="orders"
BACKUP_FILE="/backups/full_20240115_020000.sql.gz"

# Step 1: Extract just the CREATE TABLE for our table
zcat ${BACKUP_FILE} | \
  sed -n "/^-- Table structure for table \`${TABLE_NAME}\`/,/^-- /p" | \
  head -50

# Step 2: Use mysqldump directly to get just that table's backup
# (The proper way — extract from the last good full backup)
mysqldump \
  --defaults-extra-file=/root/.mybackup.cnf \
  --single-transaction \
  --no-data \
  ecommerce_db orders \
  > /tmp/orders_schema.sql

# Step 3: Extract INSERT data for orders table from backup file
# Use Python for reliable extraction (sed/awk can fail on complex SQL)
python3 << 'EOF'
import gzip, re

table = "orders"
in_table = False

with gzip.open('/backups/full_20240115_020000.sql.gz', 'rt', encoding='utf-8') as f:
    with open('/tmp/orders_data.sql', 'w') as out:
        for line in f:
            if f"-- Dumping data for table `{table}`" in line:
                in_table = True
            elif in_table and line.startswith('-- ') and table not in line:
                in_table = False
            if in_table:
                out.write(line)
print("Extraction complete")
EOF

# Step 4: Restore
mysql -u root -p ecommerce_db < /tmp/orders_schema.sql
mysql -u root -p ecommerce_db < /tmp/orders_data.sql
# Verify
mysql -u root -p -e "SELECT COUNT(*) FROM ecommerce_db.orders;"
🚨

Production Emergency Protocol: Before ANY restore, always: (1) Take a backup of current state first, (2) put the app in maintenance mode, (3) test the restore on staging first if time allows, (4) document exact commands run, (5) alert your team.

Binary Log & Point-in-Time Recovery (PITR)

PITR is the ability to restore your database to any specific point in time. It requires: a full backup as the baseline + binary logs from that point forward. This is production-grade DR — you can recover to the exact second before an incident.

Understanding Binary Logs

Binary Log Setup & Verification
# In /etc/mysql/mysql.conf.d/mysqld.cnf — enable binary logging
[mysqld]
log_bin                = /var/log/mysql/mysql-bin
binlog_format          = ROW             # ROW is default and recommended
expire_logs_days       = 14             # keep 14 days of binlogs
max_binlog_size        = 100M           # rotate when reaches 100MB
server_id              = 1              # required for binlog + replication

------------------------------------------------------------
# Check current binary log status
mysql -u root -p -e "SHOW BINARY LOGS;"
# Log_name           File_size  Encrypted
# mysql-bin.000040   104857234  No
# mysql-bin.000041   104857189  No
# mysql-bin.000042   5021023    No    ← current active log

# Check current position
mysql -u root -p -e "SHOW BINARY LOG STATUS\G"
# File: mysql-bin.000042
# Position: 5021023

PITR — Complete Walkthrough

🔥 PITR Scenario: Accidental DROP TABLE at 14:32:45 DISASTER RECOVERY

Last full backup: today at 02:00 AM. You need to recover everything up to 14:32:40 (5 seconds before the DROP).

bash · Complete PITR WalkthroughSTEP BY STEP
##############################################################
# STEP 1: Identify the incident and gather info
##############################################################
# Incident: DROP TABLE ecommerce_db.orders executed at 14:32:45
# Full backup available: /backups/full_20240115_020000.sql.gz
# Binary logs available: /var/log/mysql/mysql-bin.000042+

# Find the DROP TABLE event in binary logs to get exact position
mysqlbinlog \
  --start-datetime="2024-01-15 14:00:00" \
  --stop-datetime="2024-01-15 14:35:00" \
  /var/log/mysql/mysql-bin.000042 | grep -B5 "DROP TABLE"
# Output shows: # at 8841920 — this is the position of the DROP

##############################################################
# STEP 2: Restore the full backup
##############################################################
echo "Restoring full backup..."
zcat /backups/full_20240115_020000.sql.gz | mysql -u root -p

# Check what binlog position the full backup ended at
zcat /backups/full_20240115_020000.sql.gz | head -100 | grep "MASTER_LOG"
# -- CHANGE MASTER TO MASTER_LOG_FILE='mysql-bin.000042', MASTER_LOG_POS=4891620;
# So we need to apply binlog changes from position 4891620 onwards

##############################################################
# STEP 3: Apply binary logs UP TO (but not including) the DROP
##############################################################
mysqlbinlog \
  --start-position=4891620 \   # where our full backup left off
  --stop-position=8841920 \    # position BEFORE the DROP TABLE
  /var/log/mysql/mysql-bin.000042 \
  | mysql -u root -p

# Alternatively, use timestamps (slightly less precise but easier)
mysqlbinlog \
  --start-datetime="2024-01-15 02:00:00" \   # full backup time
  --stop-datetime="2024-01-15 14:32:40" \    # 5 seconds before incident
  /var/log/mysql/mysql-bin.000042 \
  /var/log/mysql/mysql-bin.000043 \            # apply multiple logs if needed
  | mysql -u root -p

##############################################################
# STEP 4: Verify recovery
##############################################################
mysql -u root -p -e "SELECT COUNT(*) FROM ecommerce_db.orders;"
mysql -u root -p -e "SELECT MAX(created_at) FROM ecommerce_db.orders;"
# Should show orders up to 14:32:40

echo "PITR complete. Data recovered to 2024-01-15 14:32:40"

Handling Multiple Binary Log Files

Multi-Binlog PITR
# When changes span multiple binlog files, pipe them all at once
mysqlbinlog \
  --start-position=4891620 \
  /var/log/mysql/mysql-bin.000042 \
  /var/log/mysql/mysql-bin.000043 \
  /var/log/mysql/mysql-bin.000044 \
  --stop-datetime="2024-01-15 14:32:40" \
  | mysql -u root -p

# ⚠️  IMPORTANT: Apply ALL logs in one pipe, not separately.
# Applying them one-by-one can cause FK violation issues at boundaries.

Automation, Cron & Retention

Production Backup Script

bash · Production Backup Script/etc/cron.d/mysql-backup
#!/bin/bash
# /usr/local/bin/mysql_backup.sh
# Production MySQL Backup Script
# Cron: 0 2 * * * root /usr/local/bin/mysql_backup.sh >> /var/log/mysql_backup.log 2>&1

set -euo pipefail    # exit on error, unset variable, pipe failure

# ===== CONFIGURATION =====
BACKUP_DIR="/var/backups/mysql"
CNF_FILE="/root/.mybackup.cnf"
RETENTION_DAYS=30          # keep backups for 30 days
ALERT_EMAIL="dba-team@company.com"
MIN_BACKUP_SIZE_MB=100     # backup must be at least 100MB or alert
DATE=$(date +%Y%m%d)
TIMESTAMP=$(date +%Y%m%d_%H%M%S)
DAILY_DIR="${BACKUP_DIR}/${DATE}"
LOG_PREFIX="[$(date '+%Y-%m-%d %H:%M:%S')]"

# ===== FUNCTIONS =====
log() { echo "${LOG_PREFIX} $1"; }
alert() { echo "$1" | mail -s "⚠️ MySQL Backup Alert - $(hostname)" ${ALERT_EMAIL}; }

cleanup_old_backups() {
  log "Removing backups older than ${RETENTION_DAYS} days..."
  find ${BACKUP_DIR} -type d -mtime +${RETENTION_DAYS} -exec rm -rf {} + 2>/dev/null || true
  log "Cleanup complete. Current disk usage: $(du -sh ${BACKUP_DIR})"
}

# ===== MAIN =====
log "===== Starting MySQL Backup ====="
mkdir -p "${DAILY_DIR}"

# Full backup
BACKUP_FILE="${DAILY_DIR}/full_${TIMESTAMP}.sql.gz"
log "Starting full backup → ${BACKUP_FILE}"

mysqldump \
  --defaults-extra-file="${CNF_FILE}" \
  --single-transaction \
  --source-data=2 \
  --flush-logs \
  --routines --triggers --events \
  --add-drop-database \
  --max-allowed-packet=512M \
  --all-databases \
  | pigz -9 \
  > "${BACKUP_FILE}"

BACKUP_EXIT=$?
if [ $BACKUP_EXIT -ne 0 ]; then
  alert "CRITICAL: mysqldump failed with exit code ${BACKUP_EXIT} on $(hostname)"
  exit 1
fi

# Validate backup size
BACKUP_SIZE_MB=$(du -m "${BACKUP_FILE}" | cut -f1)
if [ "${BACKUP_SIZE_MB}" -lt "${MIN_BACKUP_SIZE_MB}" ]; then
  alert "WARNING: Backup size ${BACKUP_SIZE_MB}MB is below threshold ${MIN_BACKUP_SIZE_MB}MB"
fi

# Test integrity of compressed file
pigz -t "${BACKUP_FILE}" || { alert "Backup integrity check FAILED: ${BACKUP_FILE}"; exit 1; }

# Copy binlogs from today
log "Copying binary logs..."
mysql --defaults-extra-file="${CNF_FILE}" -e "FLUSH BINARY LOGS;"
rsync -av /var/log/mysql/mysql-bin.0* "${DAILY_DIR}/binlogs/" 2>/dev/null || true

# Create MD5 checksum
md5sum "${BACKUP_FILE}" > "${BACKUP_FILE}.md5"

log "Backup SUCCESS: ${BACKUP_FILE} (${BACKUP_SIZE_MB}MB)"
cleanup_old_backups
log "===== Backup Complete ====="

Interview Questions & Answers

These are the most-asked questions in DBA interviews — from junior to principal level. Each answer is written from a senior DBA perspective with production context. Follow-up questions are included for each.

EASY Q1. What is mysqldump and what does it produce?

mysqldump is a logical backup utility included with MySQL. It connects to a running MySQL instance and produces a SQL script containing all the DDL (Data Definition Language) and DML (Data Manipulation Language) statements needed to recreate the database in its exact state — including CREATE TABLE, CREATE INDEX, INSERT statements, views, stored procedures, triggers, and events.

The output is a plain-text .sql file that is human-readable, portable across operating systems, and can be restored on any compatible MySQL version. In production, we always compress the output with gzip/pigz to reduce storage by 80–90%.

Key characteristic: It's a logical backup (SQL statements) as opposed to a physical backup (raw data files). This makes it highly portable but slower to restore at very large scale compared to tools like Percona XtraBackup.

⤷ Follow-up Questions
Follow-up: What's the difference between a logical and physical backup?
A logical backup (mysqldump) contains SQL statements — portable, human-readable, and storage-engine independent. A physical backup (XtraBackup, raw file copy) copies actual data files on disk. Physical backups are faster to create and restore for large databases, but are tied to the specific MySQL version and OS. For databases over 50GB, most production shops use physical backups for DR and mysqldump for smaller, selective restores.
Follow-up: Can mysqldump back up non-InnoDB tables consistently?
Yes, but not without locking. For MyISAM tables, mysqldump issues LOCK TABLES, which blocks writes for the duration of the table dump. This is why InnoDB is the recommended engine — with --single-transaction, it gets a consistent snapshot through MVCC without any locking.
EASY Q2. What is the --single-transaction flag and why is it critical for InnoDB?

--single-transaction is the most important flag for backing up InnoDB tables in a live production environment. When used, mysqldump issues a START TRANSACTION WITH CONSISTENT SNAPSHOT before beginning the dump. This leverages InnoDB's MVCC (Multi-Version Concurrency Control) to get a point-in-time snapshot of all data.

The critical benefit: other transactions can continue reading and writing while the backup runs. There are NO table locks held for InnoDB tables. This means zero impact on production application performance — users continue writing orders, the backup runs in the background.

Without this flag, mysqldump acquires LOCK TABLES for each table, which can block writes for the entire duration of the backup — potentially hours on a large database.

⤷ Follow-up Questions
Follow-up: Does --single-transaction work for MyISAM tables?
No. --single-transaction only provides consistent snapshots for transactional storage engines (InnoDB, XtraDB). For MyISAM tables in the same backup, mysqldump still acquires LOCK TABLES. This is one of the strongest arguments for migrating all tables to InnoDB in production.
Follow-up: Can DDL operations (ALTER TABLE) during a --single-transaction backup corrupt it?
Yes! This is a known caveat. If someone runs ALTER TABLE while --single-transaction is active, the transaction snapshot is invalidated for that table. MySQL documentation warns that "It is not possible to ensure consistent backup between InnoDB and non-InnoDB tables." In production, we use change management processes to prevent DDL during backup windows, or we use Percona XtraBackup which handles this better.
MEDIUM Q3. What is Point-in-Time Recovery (PITR) and how do you implement it with mysqldump?

PITR (Point-in-Time Recovery) is the ability to restore a database to any specific moment in time — not just to the last backup, but to within seconds of an incident. It's essential when human error (accidental DELETE/DROP) or corruption occurs between regular backup intervals.

Implementation requires two components:

1. Full backup baseline: A mysqldump taken with --source-data=2 and --flush-logs. The --source-data=2 embeds the binary log filename and position at the time of backup as a comment in the dump file. This is your recovery anchor point.

2. Binary logs: MySQL's transaction journal, which records every single data change after the backup. Binary logging must be enabled (log_bin=ON) on the server.

Recovery process: (1) Restore the full backup. (2) Use mysqlbinlog to extract and replay changes from the backup's binlog position up to a specified stop-position or stop-datetime — crucially, stopping BEFORE the damaging event.

⤷ Follow-up Questions
Follow-up: How do you find the exact binary log position of a damaging event?
Use mysqlbinlog with --start-datetime near the incident time and grep for the statement: mysqlbinlog --start-datetime="2024-01-15 14:00:00" mysql-bin.000042 | grep -B10 "DROP TABLE". The "# at NNNNNN" line before the event shows the position. You set --stop-position to that number to replay everything up to (but not including) that event.
Follow-up: What is the difference between --source-data=1 and --source-data=2?
--source-data=1 outputs an uncommented CHANGE REPLICATION SOURCE TO statement — this is used when the backup is intended to seed a replica, because it will be executed during restore automatically. --source-data=2 outputs it as a SQL comment — informational only, used when you just want to record the binlog position for PITR purposes without accidentally changing replication configuration on restore.
MEDIUM Q4. How would you back up a 500GB production MySQL database with zero downtime?

For a 500GB InnoDB database, mysqldump alone may not be the optimal primary tool due to restore time (can take 4-8+ hours), but it can still achieve zero downtime during the backup itself.

The mysqldump approach: Use --single-transaction with compression piped to pigz (parallel gzip). The backup itself has zero production impact for InnoDB. However, the dump + transfer + restore cycle for 500GB could take 6+ hours, violating a tight RTO.

The production-grade approach for 500GB+:

1. Use Percona XtraBackup for the primary DR backup — it's a physical hot backup tool that can back up 500GB in 20-40 minutes and restore in 30-60 minutes.

2. Supplement with mysqldump for selective restores (single table, schema-only, dev snapshots).

3. If MySQL version allows, use read replicas — run mysqldump against the replica (not primary) to eliminate even any theoretical performance impact on production traffic.

⤷ Follow-up Questions
Follow-up: How do you run mysqldump against a replica to reduce primary load?
Connect to the replica's IP/socket with the same backup user. On the replica, use --source-data=2 to record the replica's binlog position (which corresponds to a primary binlog position), and consider using --stop-slave-sql-thread / STOP REPLICA SQL_THREAD before the dump for a perfectly consistent point. After the dump completes, START REPLICA SQL_THREAD. This way your primary has zero backup overhead.
Follow-up: What is your RPO and RTO for this setup?
With daily mysqldump + continuous binary logging: RPO is effectively 0 (you can recover to any second), but the RTO for a full restore of 500GB with mysqldump might be 4-8 hours. With XtraBackup: RPO same (binary logs), RTO drops to 1-2 hours. For sub-minute RTO, you'd use MySQL InnoDB Cluster or Group Replication for automatic failover — backups are for catastrophic scenarios, replication handles availability.
MEDIUM Q5. A developer ran "DELETE FROM orders WHERE 1=1" in production. What do you do?

This is the classic production incident scenario. Here is the exact runbook I follow:

Immediate triage (first 2 minutes):

1. Stop the bleeding: Put the application in maintenance mode immediately to prevent further writes that overwrite binlog space or complicate recovery.

2. Do NOT restart MySQL and do NOT run any more queries that modify data.

3. Note the exact timestamp of the incident and which binary log is currently active: SHOW BINARY LOG STATUS\G

Recovery (next 30 minutes):

4. Identify the binlog position of the DELETE using: mysqlbinlog --start-datetime="..." binlog | grep -B5 "DELETE FROM orders"

5. Restore last night's full backup to a staging server first, verify it, then apply binlogs stopping just before the DELETE position.

6. Verify row counts match expectations (SELECT COUNT(*)).

7. Either: swap staging to production (if architecture allows) OR dump just the orders table from staging and restore it to production.

Post-incident: Implement SQL review gates, test user privileges (should the developer have DELETE on production?), and conduct a blameless post-mortem.

⤷ Follow-up Questions
Follow-up: What if binary logging was not enabled?
Without binary logs, you can only recover to the last full backup — all transactions since then are permanently lost. This is a catastrophic scenario. In production, binary logging must always be enabled. This is why I verify log_bin=ON as the first check in every DBA onboarding. The storage cost (usually 5-20GB/day) is trivial compared to the risk.
Follow-up: How do you prevent this from happening again?
Multiple layers: (1) Never give developers direct DELETE/DROP privileges on production — use application service accounts with minimal permissions. (2) Implement a SQL review/approval workflow for production changes. (3) Enable sql_safe_updates=1 for interactive sessions (requires WHERE clause on DELETE/UPDATE). (4) Consider MySQL's "delayed replica" — a replica that lags 1 hour — as an extra recovery layer. (5) Regular fire drills: practice restores quarterly so the team stays sharp.
MEDIUM Q6. What are RPO and RTO? How do they drive your backup strategy?

RPO (Recovery Point Objective) is the maximum amount of data loss a business can tolerate, measured in time. "We cannot lose more than 1 hour of orders" = RPO of 1 hour.

RTO (Recovery Time Objective) is the maximum acceptable downtime during a recovery event. "We must be back online within 4 hours" = RTO of 4 hours.

How they drive backup strategy:

RPO drives backup frequency: RPO of 1 hour → hourly backups or continuous binary logging. RPO of 24 hours → daily backups may suffice.

RTO drives backup type and testing: RTO of 1 hour → mysqldump alone won't work for 500GB (takes longer); use XtraBackup or replication failover. RTO of 8 hours → mysqldump restore is feasible for most databases.

In practice: I always start a backup strategy design by asking the business for their RPO and RTO. These are business decisions, not technical ones. A 1-minute RPO / 5-minute RTO requires active-active replication, not just backups.

⤷ Follow-up Questions
Follow-up: What backup strategy achieves RPO near zero?
Near-zero RPO requires continuous binary log shipping (binlog streaming) to a remote location combined with semi-synchronous replication. The full backup is your baseline; binary logs are streamed in real-time. This achieves RPO of seconds, limited only by network latency. MySQL Group Replication and InnoDB Cluster provide built-in synchronous replication for near-zero RPO with automatic failover.
MEDIUM Q7. How do you verify that a backup is valid and restorable?

This is one of the most important DBA practices — and the most neglected. There are multiple levels of verification:

Level 1 — File integrity: Check the compressed file isn't corrupted: pigz -t backup.sql.gz. Check MD5/SHA256 checksum matches the one recorded at backup time.

Level 2 — SQL validity: For uncompressed dumps, use mysql --verbose --dry-run or parse the SQL with: zcat backup.sql.gz | grep -c "^INSERT INTO" to verify INSERT statements exist.

Level 3 — Actual restore test: The only real verification. Restore the backup to a separate staging/test server and run validation queries: SELECT COUNT(*) on critical tables, verify foreign key counts, run application smoke tests.

Production practice: I run automated weekly restore tests on a dedicated restore-test server. A cron job restores Sunday's backup every Monday morning and sends a report with row counts for key tables to the DBA team. If something is wrong with the backup, we find out Monday — not during a DR event on Friday night.

⤷ Follow-up Questions
Follow-up: What should you check immediately after a restore?
1. Compare row counts on critical tables with production. 2. Verify the latest record timestamps — MAX(created_at) should reflect expected data freshness. 3. Run CHECKSUM TABLE on critical tables (works for MyISAM; for InnoDB use application-level checksums). 4. Verify stored procedures, views, triggers exist. 5. Test application connectivity and a sample transaction. 6. Check for any errors in MySQL error log (/var/log/mysql/error.log) from the restore process.
HARD Q8. How do you ensure backup consistency when you have both InnoDB and MyISAM tables?

This is a real challenge in legacy production systems. The problem: --single-transaction only works for InnoDB. MyISAM tables still need LOCK TABLES. If you use both flags, --single-transaction takes precedence for InnoDB but MyISAM tables will be locked.

Options:

1. --lock-all-tables: Issues a global FLUSH TABLES WITH READ LOCK. Everything is locked for the entire dump duration. Consistent, but blocks all writes — acceptable only in a maintenance window or for small databases.

2. Two-pass approach: First dump InnoDB tables with --single-transaction (no locks), then dump MyISAM tables with --lock-tables in a second call. Slightly inconsistent between the two passes, but may be acceptable depending on business requirements.

3. Best solution: Eliminate MyISAM. Convert all MyISAM tables to InnoDB: ALTER TABLE tablename ENGINE=InnoDB;. This is the production standard since MySQL 5.5+. MyISAM has no ACID compliance, no crash recovery, and no row-level locking anyway.

4. For the transition period, schedule MyISAM table backups during low-traffic hours and accept a short write lock.

⤷ Follow-up Questions
Follow-up: What does FLUSH TABLES WITH READ LOCK do exactly?
FTWRL closes all open tables, flushes the query cache, and acquires a global read lock that blocks all writes (INSERT, UPDATE, DELETE, DDL) for ALL tables on the entire MySQL instance. It also blocks binary log writes. It's the most aggressive lock in MySQL. In production, this can cause application timeouts if held for more than a few seconds. Always release with UNLOCK TABLES as soon as the backup position is recorded.
HARD Q9. What are the limitations of mysqldump in a large-scale production environment?

This is a senior-level question. mysqldump is excellent but has real limitations you must know:

1. Serial execution: mysqldump is single-threaded. A 200GB database might take 2-3 hours to dump and 4-6 hours to restore — all sequentially. Tools like mydumper/myloader parallelize this significantly.

2. Restore performance: Replaying millions of INSERT statements is slow. It's not uncommon for a 50GB dump to take 90 minutes to restore. During restore, indexes are rebuilt one-by-one.

3. No incremental backups natively: mysqldump always does a full dump. True incrementals require binary log management separately.

4. Memory pressure during restore: Large INSERT batches (--extended-insert) can cause buffer pool pressure.

5. No streaming/online backup for MyISAM: MyISAM tables require locks, causing production impact.

6. GTID complications: Restoring a dump on a GTID-enabled replica requires careful handling of SET @@GTID_PURGED — a common source of replication setup errors.

Alternatives at scale: Percona XtraBackup (physical, fast), mydumper/myloader (parallel logical), MySQL Enterprise Backup, or cloud-native solutions (RDS automated backups, Aurora's continuous backup).

⤷ Follow-up Questions
Follow-up: How does mydumper improve on mysqldump?
mydumper is a parallel multi-threaded dump tool. It uses multiple threads (--threads=8) to dump multiple tables simultaneously, dramatically reducing dump time. myloader (the restore counterpart) similarly restores in parallel. On a 200GB database with 8 threads, what takes 3 hours with mysqldump takes 45 minutes with mydumper. It also supports more granular consistency options and produces chunk files rather than one monolithic SQL file, allowing for selective restores without parsing.
Follow-up: How do you handle GTIDs when restoring a mysqldump?
mysqldump includes SET @@GTID_PURGED='uuid:1-N' when backing up a GTID-enabled server. When restoring, this tells MySQL "these transactions already happened, don't expect them." On a fresh instance, this is fine. On a server that already has GTID history, you'll get an error. Solution: use --set-gtid-purged=OFF in mysqldump if you're doing a partial restore or restoring to a server already in a replication topology. Use =ON (default) for fresh replica seeds.
HARD Q10. How do you design a complete backup strategy for a mission-critical production MySQL environment?

This is the principal/staff DBA question. A complete strategy has multiple layers:

Layer 1 — High Availability (handles most "outages"): MySQL InnoDB Cluster or Group Replication with 3+ nodes across availability zones. Automatic failover in 30-60 seconds. This handles server failure, not data corruption.

Layer 2 — Delayed Replica: One replica configured with CHANGE REPLICATION SOURCE TO SOURCE_DELAY=3600 (1 hour delay). If a bad query is detected within 1 hour, you can stop this replica and recover from it without touching backups.

Layer 3 — Continuous Binary Log Archival: mysqlbinlog streaming to S3/object storage in real-time. Achieves near-zero RPO independently of backup schedule.

Layer 4 — Daily Physical Backup (XtraBackup): Full physical backup nightly, stored locally + offsite (S3 with cross-region replication). Retention: 30 days local, 1 year offsite.

Layer 5 — Weekly mysqldump: For portability, long-term archiving, and selective restore capability. Retained for 1 year.

Layer 6 — Testing: Weekly automated restore tests. Monthly full DR drill with RTO measurement. Quarterly "chaos day" where team practices restores under pressure.

Layer 7 — Monitoring: Alert on backup failures within 15 minutes. Alert on backup age > 26 hours. Alert on binlog gap. Dashboard showing last successful backup time, size, and restore test results.

⤷ Follow-up Questions
Follow-up: What's the "3-2-1 backup rule" and how do you apply it to MySQL?
The 3-2-1 rule: 3 copies of data, on 2 different media types, with 1 offsite copy. For MySQL: (1) Live production database, (2) Local backup server with mysqldump/XtraBackup files, (3) Offsite copy in cloud storage (S3, GCS) in a different region. The two media types could be: SSD/NVMe for live + HDD for local backup + cloud object storage for offsite. This ensures that no single failure (hardware, datacenter fire, ransomware) can destroy all copies.
Follow-up: How do you handle backup encryption for compliance?
For HIPAA/PCI-DSS compliance, backups must be encrypted at rest and in transit. Encrypt the stream before writing: mysqldump ... | gzip | openssl enc -aes-256-cbc -pass file:/root/.backup_key > backup.sql.gz.enc. Store the encryption key in a secrets manager (HashiCorp Vault, AWS KMS) — never on the same server as the backups. For cloud storage, use server-side encryption (S3 SSE-KMS). Document the key management process as part of your DR runbook — an encrypted backup without the key is worthless.
EASY Q11. How do you restore a single table from a full mysqldump backup?

This is one of the most common real-world DBA tasks. Since mysqldump puts everything in one file, you need to extract the specific table's section. There are a few methods:

Method 1 — Use sed to extract the section: Find the CREATE TABLE and INSERT block for your table using sed with address ranges matching the table's comment headers that mysqldump includes.

Method 2 — Re-dump just that table: If the database is still accessible (e.g., data was corrupted, not deleted), simply run mysqldump again for just that table: mysqldump ... dbname tablename > table_backup.sql

Method 3 — Python script: For reliability on large files (100GB+), use Python to parse line-by-line and extract the relevant INSERT statements.

Best practice: For critical tables (orders, users, payments), always keep separate per-table backups in addition to the full backup. This makes selective restores trivial.

⤷ Follow-up Questions
Follow-up: What structure does mysqldump use to separate tables in the output file?
mysqldump uses standard SQL comments as delimiters: "-- Table structure for table `tablename`" precedes the CREATE TABLE statement, and "-- Dumping data for table `tablename`" precedes the INSERT statements. These comment markers are consistent and reliable for parsing. The sections are also separated by "-- ------" divider lines. You can grep for these patterns to find byte offsets for efficient extraction without reading the whole file.
MEDIUM Q12. What privileges does a backup user need? Why should you not use root for backups?

Using root for automated backups violates the principle of least privilege — one of the most fundamental security principles. If the backup script or credential file is compromised, an attacker gets full root access to your database.

Minimum required privileges for a backup user:

• SELECT — read all table data

• SHOW VIEW — dump view definitions

• TRIGGER — dump trigger definitions

• LOCK TABLES — needed for MyISAM tables (can skip if pure InnoDB + --single-transaction)

• EVENT — dump scheduled events

• RELOAD — needed for FLUSH TABLES WITH READ LOCK and FLUSH BINARY LOGS

• REPLICATION CLIENT — needed for SHOW BINARY LOG STATUS (for --source-data)

• PROCESS — needed to see all threads (some mysqldump versions require this)

⤷ Follow-up Questions
Follow-up: How do you securely store backup credentials?
Never hardcode passwords in scripts or cron entries. Use a MySQL option file (~/.mybackup.cnf or /etc/mysql/backup.cnf) with [client] section, chmod 600 so only the backup user can read it, and use --defaults-extra-file= to reference it. For cloud environments, use AWS Secrets Manager, HashiCorp Vault, or GCP Secret Manager to inject credentials at runtime. Rotate backup user passwords every 90 days as part of your security policy, using a secrets manager to avoid manual script updates.
HARD Q13. Explain what happens step-by-step when you run mysqldump --single-transaction --all-databases internally.

This tests deep internal knowledge. Here's the exact sequence:

1. mysqldump connects to MySQL server.

2. It sets the session transaction isolation level: SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ — required for consistent MVCC snapshot.

3. If --flush-logs specified: FLUSH LOGS — rotates binary log.

4. If --source-data specified: FLUSH TABLES WITH READ LOCK — brief global lock to get a consistent binlog position.

5. Records binary log position: SHOW BINARY LOG STATUS — saves filename and position.

6. Issues: START TRANSACTION WITH CONSISTENT SNAPSHOT — opens the MVCC snapshot (InnoDB).

7. Releases the global lock: UNLOCK TABLES — other writes can now proceed.

8. Queries information_schema for all databases and tables.

9. For each table: issues SELECT /*!40001 SQL_NO_CACHE */ * FROM tablename and writes CREATE TABLE + INSERT statements.

10. Writes stored routines, events, triggers for each database.

11. Transaction is committed/closed when dump completes. Total lock time = step 4 to step 7 (usually milliseconds).

⤷ Follow-up Questions
Follow-up: Why is REPEATABLE READ isolation important for the dump?
REPEATABLE READ combined with START TRANSACTION WITH CONSISTENT SNAPSHOT means all reads within the transaction see the same consistent state of the database from the moment the transaction started. Even if rows are inserted or updated in other sessions while the dump is running (which could take hours for large databases), the dump sees the original consistent state. This is InnoDB MVCC at its finest — it's what makes --single-transaction truly non-blocking.
MEDIUM Q14. What is the difference between --flush-logs and FLUSH BINARY LOGS? When do you use each?

--flush-logs is a mysqldump flag that triggers a binary log rotation (FLUSH LOGS) at the very start of the backup process. This creates a clean boundary: the new binlog file starts exactly when the backup starts. This simplifies PITR because you know: "to replay changes after this backup, just apply binlog files starting from mysql-bin.XXXXXX+1."

FLUSH BINARY LOGS is a MySQL SQL statement you run directly that rotates the current binary log file to a new one. Useful for: manually archiving current binlogs, reducing the size of individual binlog files, creating clean checkpoints before maintenance operations.

When to use --flush-logs in mysqldump: Always when creating a PITR-capable backup alongside --source-data. The combination gives you: (1) a backup, (2) the exact binlog file where post-backup changes begin, (3) a clean binlog file to archive with the backup.

Caution: If you have multiple databases and use --flush-logs without --lock-all-tables or --single-transaction, the flush happens once at the start. Databases dumped later in the process still have correct PITR capability because the binlog position in the dump reflects the snapshot position, not the flush position.

HARD Q15. You need to migrate a 200GB MySQL 5.7 database to MySQL 8.0. What is your process?

This is a real senior DBA scenario combining backup, testing, and version management:

Phase 1 — Preparation (1-2 weeks before): Set up a MySQL 8.0 test server. Run mysqlcheck -u root -p --all-databases on the 5.7 instance to check for issues. Review MySQL 8.0 breaking changes (removed NO_AUTO_CREATE_USER, utf8mb3 deprecation, reserved words). Test your application against MySQL 8.0 on the test server.

Phase 2 — Test migration: Take a mysqldump of 5.7: mysqldump --single-transaction --routines --triggers --events --all-databases > migration_test.sql. Restore to 8.0 test server. Run mysql_upgrade if needed. Run application smoke tests. Document all issues found.

Phase 3 — Production migration (maintenance window): (1) Announce maintenance window. (2) Stop application. (3) Take final mysqldump from 5.7: mysqldump --single-transaction --source-data=2 --all-databases | pigz > final_migration.sql.gz. (4) Restore to MySQL 8.0. (5) Run ANALYZE TABLE on all tables (statistics format changed in 8.0). (6) Test critical queries. (7) Point application to new server. (8) Monitor closely for 24-48 hours.

Rollback plan: Keep the 5.7 server running and accessible. If 8.0 has critical issues within the first hour, repoint application DNS back to 5.7. You can't roll backward from 8.0 to 5.7 with the data files directly — this is why the 5.7 server must stay live until you're confident.

⤷ Follow-up Questions
Follow-up: What are the main breaking changes between MySQL 5.7 and 8.0 that affect mysqldump restores?
Key breaking changes: (1) utf8 is now an alias for utf8mb3 with deprecation warnings — collation names changed. (2) Several SQL modes are stricter by default (STRICT_TRANS_TABLES). (3) Authentication default changed from mysql_native_password to caching_sha2_password — application connectors must support it. (4) GROUP BY behavior changed — queries with GROUP BY that worked in 5.7 may fail in 8.0 strict mode. (5) Reserved words expanded — if you have column/table names that are now reserved words, you'll get syntax errors during restore. The fix: use backtick quoting (mysqldump does this by default). (6) The mysql system database has new tables; restoring --all-databases can conflict — use --databases to exclude it and let MySQL 8.0 manage its own system tables.
MEDIUM Q16. What is the --hex-blob option and when would you use it?

--hex-blob dumps binary data (BLOB, BINARY, VARBINARY, BIT columns) as hexadecimal strings instead of raw binary. For example, instead of unreadable binary characters in the INSERT statement, you get: INSERT INTO t VALUES (0x89504E470D0A1A0A...).

When to use it:

1. When your binary data contains null bytes (\0) or other control characters that break SQL parsing during restore.

2. When transferring dumps through systems that may mangle binary data (some FTP clients, email systems, text editors).

3. When you need to diff two backups — hex representation is diff-friendly.

4. When storing images, documents, or encrypted data in BLOB columns — the hex representation is always ASCII-safe.

Trade-off: Hex encoding doubles the size of binary data in the dump file. A 1MB BLOB becomes 2MB in the SQL file. For databases heavy with BLOBs, this can significantly increase dump size. In most modern architectures, storing files in databases is discouraged — use S3/object storage and store just the URL in MySQL.

HARD Q17. How do you handle mysqldump for databases with foreign key constraints during restore?

Foreign key constraints are a common source of restore failures. mysqldump handles this by default, but you need to understand the mechanism:

What mysqldump does automatically: It adds SET FOREIGN_KEY_CHECKS=0 at the beginning of the dump and SET FOREIGN_KEY_CHECKS=1 at the end. This allows INSERT statements to be applied in any order without FK violation errors — critical because mysqldump dumps tables alphabetically, not in FK dependency order.

When this can fail: If you're doing a partial restore (only some tables from a dump), you might restore a child table's data without the parent table existing, which will fail when FK checks are re-enabled. Solution: restore parent tables first, then child tables, or keep FK checks off for the restore session.

Best practice for partial restores with FKs:

mysql -u root -p << 'EOF'
SET FOREIGN_KEY_CHECKS=0;
source /tmp/partial_restore.sql;
SET FOREIGN_KEY_CHECKS=1;
-- Verify FK integrity after restore:
SELECT * FROM information_schema.KEY_COLUMN_USAGE 
WHERE REFERENCED_TABLE_NAME IS NOT NULL;
EOF
⤷ Follow-up Questions
Follow-up: What does SET UNIQUE_CHECKS=0 do during restore and should you use it?
SET UNIQUE_CHECKS=0 tells InnoDB to skip unique index verification during INSERT operations. mysqldump adds this by default along with FK_CHECKS=0. It speeds up restores significantly on large tables with many unique indexes. The assumption is: if the data was consistent when backed up, it's safe to skip uniqueness verification during restore. You should keep it (don't remove it from dumps) — it can 2-3x your restore speed on large tables. Always ensure the source data was consistent before relying on this optimization.
EASY Q18. What is the difference between mysqldump --databases and specifying the database name directly?

This is a subtle but practically important distinction:

Without --databases (e.g., mysqldump dbname > backup.sql): The dump does NOT include CREATE DATABASE or USE dbname statements. The output contains only table-level DDL and INSERT statements. To restore, you must first create the database and specify it during restore: mysql dbname < backup.sql. This format is useful when migrating to a differently-named database.

With --databases (e.g., mysqldump --databases dbname > backup.sql): The dump includes CREATE DATABASE IF NOT EXISTS dbname and USE dbname statements. You can restore with just mysql < backup.sql — no need to pre-create the database. This is the safer option for full DR restores.

--all-databases implies --databases for all databases — always includes CREATE DATABASE for each.

Production recommendation: Always use --databases or --all-databases for automated backups so restores are self-contained and don't require manual database creation steps that are easy to forget under pressure.

HARD Q19. How do you monitor and alert on backup failures in production?

Backup monitoring is as important as the backups themselves. A silent backup failure can go undetected for days until you need the backup for a DR event — the worst possible time to discover it failed.

Multi-layer monitoring approach:

1. Exit code monitoring: Capture $? after mysqldump and alert immediately if non-zero. This catches network errors, auth failures, disk full conditions.

2. File size monitoring: Compare today's backup size to a baseline (yesterday's size ± 20%). A backup that's 0 bytes or 90% smaller than usual suggests a problem. Alert the team.

3. Backup age monitoring: A cron job runs at 4 AM and checks: "Is there a backup file from the last 26 hours?" If not, alert. This catches cases where the backup cron itself failed silently.

4. Integrity check: After every backup: pigz -t backup.sql.gz || alert "Backup integrity failed"

5. Restore test monitoring: Weekly automated restore test sends a pass/fail report. Track this on a dashboard.

6. Centralized alerting: PagerDuty/OpsGenie for critical alerts (backup failed = P1), email for warnings (backup slightly small = P3). Never rely on email alone for critical database backup failures.

Follow-up: What metrics would you track on a DBA backup dashboard?
Last successful backup timestamp and time elapsed since then. Backup size trend over time (detects data growth issues). Backup duration trend (growing duration = growing database or performance issues). Restore test last run date and result. Binary log gap (is archival keeping up?). Disk space remaining on backup storage (alert at 80% full). Compression ratio (sudden change may indicate backup content issues). Number of databases and tables in latest backup (sudden decrease may indicate silent data loss).
MEDIUM Q20. What is the impact of mysqldump on a running production system? How do you minimize it?

Even with --single-transaction, mysqldump is not zero-impact on production:

Actual impacts:

1. CPU: mysqldump issues a continuous stream of SELECT queries. On a busy server, this adds query load and competes for CPU with application queries. Compression (piped gzip/pigz) adds significant CPU overhead on the backup server.

2. I/O: Reading all data from disk puts pressure on the buffer pool. Fresh data pages are evicted from InnoDB buffer pool to make room for table scan pages — this can cause a "cold cache" effect where your application queries become slower because their data was evicted.

3. Network: If backing up remotely, saturates the database network interface.

4. Brief global lock (--source-data): FLUSH TABLES WITH READ LOCK for typically < 1 second to record binlog position.

Mitigation strategies:

1. Time it right: Run backups during lowest traffic period (2-4 AM for most businesses).

2. Backup from replica: Run mysqldump against a read replica to eliminate primary impact entirely.

3. Nice level: Run mysqldump with reduced priority: nice -n 19 mysqldump ...

4. ionice: Reduce I/O priority: ionice -c 3 mysqldump ... (idle class)

5. innodb_buffer_pool_size: Ensure buffer pool is large enough (70-80% of RAM) so the backup scan doesn't fully evict application data.
